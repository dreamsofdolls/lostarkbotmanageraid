# Changelog

All notable changes to this project will be documented in this file. Dates use the local calendar of the commit.

## 2026-04-22

### Added

- **`/raid-set` roster field - chained autocomplete fixes Discord 25-entry cap overflow.** Traine flag char nhiều quá chọn không hết với user có 5+ roster (5 × 6 char = 30 chars, top-25 by iLvl desc silently cuts off 5 lower-iLvl chars). Giải pháp: thêm `roster` option REQUIRED đặt FIRST trong command schema, chained autocomplete giống pattern `/remove-roster`: pick roster trước → character autocomplete chỉ show char trong roster đó. Mỗi roster max 6 char (enforced trong /add-roster) nên luôn fit 25-cap. **Option order**: `roster` (required, first) → `character` (required) → `raid` (required) → `status` (required) → `gate` (optional). Discord requires required-before-optional, roster first mới match chained autocomplete UX. **Implementation**: add `autocompleteRaidSetRoster` (list accounts với `📁 name · N chars` label, copy pattern từ `autocompleteRemoveRosterRoster`), update `autocompleteRaidSetCharacter` đọc upstream roster value filter accounts, update `autocompleteRaidSetRaid` + `autocompleteRaidSetStatus` pass rosterInput khi resolve char, update `handleRaidSetAutocomplete` dispatcher thêm roster case. **Disambiguation**: `findCharacterInUser(doc, charName, rosterName?)` accept optional roster param - when provided, scope lookup to that account's chars; khi null (text-monitor parser path), fallback first-by-iteration backwards-compatible. `applyRaidSetForDiscordId({...rosterName})` accept + forward; `handleRaidSetCommand` read roster option + pass vào apply helper. **Breaking change**: slash command schema changed, Railway redeploy sẽ re-register với required roster. Existing scripted `/raid-set character:X raid:Y status:Z` calls sẽ fail at Discord level (missing required option) - users must pick roster first. Same-named chars across rosters (e.g. main + alt cùng tên `Clauseduk`) không còn collide nhờ roster-scoped lookup. HELP_SECTIONS + README `/raid-set` section updated để describe chained autocomplete + disambiguation.

### Added

- **`/raid-check` user filter dropdown + 2 rosters per page chunking.** Traine: "dropdown chọn user discord, sau đấy hiển thị 2 roster mỗi page... Một cái dropdown ở dưới các nút bấm". Hybrid design: user-filter `StringSelectMenu` ở action row 2 (dưới Prev/Next/Sync), chunked pagination 2 rosters per embed page. **Dropdown**: first option `🌐 All users (N pending)` reset filter, tiếp theo top-24 users sort pending desc `👤 displayName (N pending)` (Discord cap 25 options). Selection → recompute pages chỉ chứa rosters của user đó, reset currentPage=0. `default: true` trên selected option preserve state qua Prev/Next clicks. **Chunking**: mỗi page stack 2 roster sections. Section = non-inline header field (name = `📁 accountName (displayName)`, value = `N pending · 🔄<relative>` - cả 2 dòng có content) + inline char cards 2-col với spacer pattern. User có 4 rosters → 2 pages (2+2). User có 3 rosters → 2 pages (2+1). Rosters consecutive giữ user grouping. **Collector handles both buttons + select menu**: bỏ `componentType: ComponentType.Button` constraint, accept all component interactions. Dispatch theo customId prefix (`raid-check-page:*` = navigation, `raid-check-filter:user` = filter change, `raid-check:sync:*` = route to global handler). **Empty filter edge case**: khi user pick filter mà user đó không còn pending char → hiện success embed `✅ username không có char nào pending trong <raid>`, preserve dropdown để user chọn lại. **Field count budget**: max scenario 2 × 6-char roster = 2 headers + 18 char/spacer inline = 20 fields, fit 25-cap. `src/raid-command.js` (handleRaidCheckCommand rewrite ~100 LOC: chunkRosters, filterByUser, addRosterSection, buildRaidCheckPage chunk-based, buildFilterDropdown, buildComponents array, collector với filter state tracking), HELP_SECTIONS + README `/raid-check` section rewritten với chunked + dropdown model.
